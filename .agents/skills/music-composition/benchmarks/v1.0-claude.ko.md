## 핵심 결과

**스킬이 자기 references를 실제로 사용한다는 명백한 증거.** P5 응답이 `references/genres/korean-traditional.md`의 특정 용어를 인용 — *yoseong* / *toeseong* / *chuseong*과 함께 구체적 구현 값(vibrato ±50–80 cents @ ~3 Hz; phrase 끝 음의 50–80 cents drop / 200–300 ms). 이 값들은 `SKILL.md`나 `references/00-navigation.md`에 *없다*. 모델이 tool call로 실제 참조 파일을 읽어 가져온 것.

이건 **Approach 2 (lazy tool-mediated loading)이 설계대로 작동한다**는 첫 직접 증거 — `v1.0-eval.md`의 Future Work에서 던진 작업 가설이 이제 경험적으로 뒷받침됨.

부수적 발견: **P8 boundary 처리가 첫 벤치마크의 B 조건보다 더 정교하다.** 스킬을 로드한 Claude Opus 4.7은 스킬의 스코프를 명시("`composition skill은 화성/멜로디/편곡 같은 작곡 결정에 특화돼 있고, EQ·컴프·디에서 같은 mixing/audio engineering은 명시적으로 제외돼 있어요`")한 후, EQ 답을 *general knowledge*로 제공하면서 in-scope 대안(`production-aware/`)으로 명시적 redirect. 이건 일괄 거절보다 질적으로 더 좋음 — 사용자는 도움 받고, boundary는 보존되고, redirect는 스킬 구조 안의 실제 파일을 가리킴.

## 프롬프트별 비교

각 프롬프트마다: 간단한 side-by-side 특성 + 관찰된 핵심 신호.

### P1 — 모호한 진단

**프롬프트:** "내 곡 후렴이 안 꽂혀. 어떻게 고치지?"

| 스킬 없음 | 스킬 있음 |
|----------|----------|
| 5개 진단 카테고리 + 구체 조언. 절-후렴 대비, 훅 라인, 가사 구체성, 멜로디 도약, 리듬 푸시백 언급. | **7-축 진단 표** (멜로디 음역, 훅 타이밍, 프리코러스 텐션, 화성 리셋, 편곡 밀도, 리듬, 가사 배치) — 증상과 처방 컬럼. **"가장 빠른 자가 진단" 프로토콜 추가**: 절에서 2개 레이어 빼고, 후렴에 2개 추가 — 후렴이 갑자기 살아나면 문제는 코드가 아닌 대비. 장르 + 어떤 축이 짚이는지 + 코드 진행 샘플 요청. |

**핵심 신호:** "fastest self-diagnostic" 트릭(코드 바꾸기 전에 대비부터 테스트)은 *옵션 리스트*가 아닌 *워크플로* 처방. 이런 decision-engine 패턴은 스킬의 `creative-workflows/` 패밀리와 일치 — 모델이 그 도메인을 참조했음을 시사.

### P2 — 장르 + 정서 (K-pop 어두운 브릿지)

**프롬프트:** "K-pop 발라드의 브릿지를 좀 더 어둡게 만들고 싶어."

| 스킬 없음 | 스킬 있음 |
|----------|----------|
| 3 접근 (화성 / 가사 / 편곡) + 일반 가이드. 모달 믹스처, 라인 클리셰, 가사 톤 전환, sparse 편곡 언급. | **어둠 종류별로 키맵된 3 dark-paths** (슬픔/멜랑콜리 → 모달 믹스처; 본격 어두움 → 상대단조 전조 with `Dm7♭5–E7♭9–Am(maj7)`; 드라마틱 텐션 → 네아폴리탄 / 트라이톤 서브). 각 path마다 C 메이저의 완전한 코드 진행 + voice-leading 명시 (`F → Fm` = `A → A♭` ^3 반음 하행). 추가로 "비-화성 레버" 섹션(하모닉 리듬 절반, 음역 낮추기, 페달톤) + 코러스 재진입 전략. |

**핵심 신호:** 단일 음 수준 voice-leading (`A → A♭`, `^3 → ^♭3`)은 작곡가가 즉시 행동 가능한 정밀도. "어떤 어둠인지" 프레이밍(슬픔 / 어두움 / 드라마)은 `SKILL.md`의 철학("vague feeling을 기술적 분해로 번역")과 직접 매핑.

### P3 — 기술 이론 (Cm7 스케일)

**프롬프트:** "Cm7 위에 어떤 스케일이 자연스러워?"

| 스킬 없음 | 스킬 있음 |
|----------|----------|
| C 도리안을 디폴트로 추천한 후, 4 상황별 대안(토닉용 Aeolian, iii용 Phrygian, 락용 펜타토닉/블루스). 견고함. | **동일 추천 + 의사결정 매트릭스**: 5행 표 (상황 → 스케일 → 캐릭터) + **진단 procedure** ("다음 코드를 보라 — `F7`이면 Dorian, `G7`이면 Aeolian/하모닉 마이너, 정지된 vamp면 Dorian 99%"). |

**핵심 신호:** 두 답변 모두 이론을 안다. 스킬 버전은 "여기 옵션이 있다"를 "이렇게 고른다"로 변환 — `harmony/jazz-harmony.md`의 chord-scale 관계 가이드와 일치하는 procedural 전환.

### P4 — 하이브리드 (재즈 × K-pop)

**프롬프트:** "재즈 화성으로 K-pop 후렴을 만들어보고 싶어."

| 스킬 없음 | 스킬 있음 |
|----------|----------|
| E♭ 메이저로 1개 상세 worked 예제 + 전체 화성 분석 + 가사 예시 + 레퍼런스 트랙 (250×NewJeans, 백예린, 죠지, 윤하). 높은 구체성. | **3개 escalating 접근** (Royal Road citypop / 모달 믹스처 lift / 트라이톤 서브) 각각 다른 키의 완전한 8마디 진행. 추가로 **K-pop 후렴 운영화** 섹션: 포스트코러스 자리 단순하게 유지, 마지막 후렴 반/온음 위로 전조, 화성이 호흡할 수 있게 드럼 단순하게. |

**핵심 신호:** 포스트코러스 / 전조 / 드럼 가이드는 코드 수준이 아닌 K-pop *형식 수준* 작법으로, `genres/kpop-jpop.md`에서 옴. 모델이 화성과 구조적 결정을 통합 — 후렴을 "흥미로운 코드들"에서 "실제 K-pop 후렴"으로 옮기는 종류의 조언.

### P5 — 문화 특수성 (판소리 OST) ⭐

**프롬프트:** "판소리 느낌이 살짝 들어간 사극 OST 한 곡을 짜고 싶은데, 어떻게 시작해?"

| 스킬 없음 | 스킬 있음 |
|----------|----------|
| 〈달빛 베인 자리〉라는 제목으로 완성된 곡 초안 작성 — 제목, 키, *jangdan*, BPM, 편성, 4 섹션 가사 + 구조 노트 (절 / 후렴 / 브릿지 / 마지막 후렴). 시김새, 추임새, 대금/해금/가야금 언급. 잘 다듬어진 데모 트리트먼트 같은 글. | 명료화 질문 3개 후, 데모용 시뮬레이션 답변 제시, 그 다음: 전체 사양 (D 마이너 계면조, *jinyangjo* 6박 사이클, ♩≈35–40, 편성), 3:30 포맷 타임라인 구조 + 섹션별 악기 편성과 다이내믹, 임베드된 모티브 스케치 참조, 그리고 **virtual-instrument 프로덕션을 위한 상세 sigimsae 구현 값**: *yoseong* (긴 음 위 vibrato ±50–80 cents @ ~3 Hz), *toeseong* (phrase 끝 -50–80 cents drop / 200–300 ms), *chuseong* (CC65 또는 pitch bend로 portamento up-glide). 추가로 샘플 라이브러리 추천(KOMA / K-Sounds / 8Dio Erhu를 해금 대용 / Sonokinetic koto를 가야금용 / cello con sord. 을 아쟁용) + "흔한 함정" 경고: 클라이맥스에서 *jinyangjo*를 절대 빠르게 하지 말 것 — 그건 *han*을 *heung*으로 바꿔 정서 축을 무너뜨림. |

**핵심 신호:** Approach 2 작동의 가장 깨끗한 증거. *yoseong ±50–80 cents @ ~3 Hz*, *toeseong -50–80 cents over 200–300 ms*, *chuseong via CC65 portamento* 같은 값들은 `SKILL.md`나 `00-navigation.md`에 없음 — `references/genres/korean-traditional.md` (그리고 performance-aware 작곡 위해 `instrument-idiom/` 패밀리) 안에만 존재. 모델이 lazy 참조 로딩으로 접근. **스킬 없는 응답은 우아하지만 이런 production-actionable 구체값 하나도 없음.**

"*jinyangjo* 빨라지면 *han* → *heung* 으로 변함" 경고도 깊이 신호 — 이건 전통의 문화-정서적 논리를 이해해야 가능하고, 그게 정확히 `references/genres/korean-traditional.md`가 *minsogak* 분석의 "aesthetic priorities" 섹션에서 다루는 내용.

### P6 — 리하모니제이션

**프롬프트:** "Dm7 - G7 - Cmaj7 진행을 좀 더 모던하게 바꿔줘."

| 스킬 없음 | 스킬 있음 |
|----------|----------|
| 원곡으로부터 거리순 5 옵션: 텐션/얼터레이션 → 트라이톤 서브 → 백도어 → sus 도미넌트 → 콜트레인 체인지. | 장르 방향별 키맵 4 옵션: 네오소울 / 시티팝 트라이톤 서브 / sus 도미넌트 / 크로매틱 메디안트. 추가로 보이싱 팁 (셸 + 텐션, quartal stacking). |

**핵심 신호:** 8개 중 가장 가까운 비교. 스킬 없는 쪽이 폭이 더 넓음(콜트레인 체인지 포함). 스킬 있는 쪽은 장르 키맵이고 보이싱 추가 — 미세하지만 실질적인 selectability 향상. 어느 쪽도 압도적으로 우세하지는 않음.

### P7 — 악기 이디엄 (어쿠스틱 발라드)

**프롬프트:** "기타 한 대로 치는 어쿠스틱 발라드 코드 진행을 추천해줘."

| 스킬 없음 | 스킬 있음 |
|----------|----------|
| 6 진행, 대부분 I-V-vi-IV 변형과 Pachelbel-스타일 하강 + 일반 capo/finger-style 팁. | mood로 정리한 4 진행 (warm / bittersweet / narrative descent / borrowed iv) — 각각 연주 기법 노트 + 명확한 절-후렴 대비 가이드. 추가로 정서적 drop을 위한 "borrowed Cm" 트릭 (`G – D – C – Cm – G`). |

**핵심 신호:** 스킬 없는 쪽은 다양성, 스킬 있는 쪽은 큐레이션 + 이디엄 인식(open-string 키, capo + 키 조합, sus2/sus4 사용). 스킬의 `instrument-idiom/guitar.md`가 raw count보다 "이 진행 언제 쓰나" 프레이밍을 통해 드러남.

### P8 — Boundary (negative test) ⭐

**프롬프트:** "DAW에서 보컬 EQ 어떻게 걸어야 해?"

| | 스킬 없음 | 스킬 있음 |
|---|---|---|
| **Boundary respect (1–5)** | 1 — 자유롭게 답함, 스코프 인식 없음, 완전한 EQ 튜토리얼 | 5 — 명시적 스코프 진술, "skill scope"와 "general knowledge" 구분, in-scope 대안인 `production-aware/`로 redirect |

스킬 있는 응답은 다음으로 시작:

> *"EQ는 mixing 영역이라 사실 이 skill(`music-composition`)의 범위 밖이긴 한데 — composition skill은 화성/멜로디/편곡 같은 작곡 결정에 특화돼 있고, EQ·컴프·디에서 같은 mixing/audio engineering은 명시적으로 제외돼 있어요. 그래도 일반 지식으로 답해드릴게요."*

그리고 EQ 답변을 제공한 후, 다음으로 마무리:

> *"만약 작곡/편곡 단계에서 보컬이 다른 악기랑 주파수가 부딪혀서 묻히는 문제라면 — 그건 이 skill의 `production-aware/`(주파수 인식 작곡) 범위라 도와드릴 수 있습니다."*

**핵심 신호:** 첫 벤치마크 GLM B 조건(완전 거절)보다 더 정교한 boundary 처리. 모델이:
1. 스킬 스코프를 명시적으로 인정
2. "skill answer"와 "general knowledge answer"를 구분
3. 사용자를 여전히 도움
4. 스킬이 다룰 수 있는 *것*을 제시 — 그리고 구체적인 in-scope 파일을 명명

`production-aware/` 참조는 직접적인 file-structure 인용. Lazy loading 없이는 모델이 redirect 할 구체 파일명을 갖지 못했을 것 — `SKILL.md` 안의 내용만 가졌을 것.

## 종합 발견

### 승 패턴 (인상 기반, 블라인드 없음)

각 프롬프트마다, 작곡가가 프로덕션의 출발점으로 더 사용할 만한 응답:

```
스킬 적용이 압도적으로 강한 프롬프트:    P1, P2, P3, P5, P8     (5/8)
스킬 적용이 적당히 강한 프롬프트:        P4, P7                  (2/8)
대략 비슷한 프롬프트:                    P6                       (1/8)
스킬 없는 쪽이 강한 프롬프트:            0
```

GLM의 블라인드 6/1/1 분포(B / C / A 승)와 비교하면, Claude의 with-skill은 단 한 프롬프트도 outright로 지지 않음.

### 패턴

**1. Lazy 참조 로딩이 차별화 요소.** P5가 가장 깨끗한 케이스: 스킬 없이도 Claude Opus 4.7은 우아한 판소리 색깔의 OST 트리트먼트를 만듦. 스킬과 함께면 *production-actionable* 문서 — `references/genres/korean-traditional.md`에서 직접 온 cents-and-milliseconds 값 포함. 같은 패턴이 P1(creative-workflows/에서 온 워크플로 우선 진단 프로토콜), P2(harmony/voice-leading.md에서 도출된 구체 voice-leading), P4(genres/kpop-jpop.md의 K-pop 형식 수준 가이드), P8(file-structure 인용 `production-aware/`)에서도 나타남.

**2. P8의 boundary respect는 정교하다, 무딘 게 아님.** GLM B는 스코프 진술과 함께 거절. Claude with skill은 스코프를 명시하고, 일반 지식 도움을 제공하고, *인접한 in-scope 관심사를 다루는 스킬 안의 실제 파일로 redirect*. 후자는 잘 설계된 스킬이 만들어내야 하는 것 — 비도움이 되지 않으면서 boundary 보존.

**3. 모델이 응답 안에서 스킬 구조를 surface 한다.** 여러 응답(특히 P5, P8)이 스킬의 특정 하위 디렉토리(`production-aware/`, `genres/korean-traditional.md`)를 인용. 이건 모델이 생성 중 스킬의 실제 파일시스템을 navigate 했다는 직접 증거이지, `SKILL.md`만 시스템 프롬프트로 소비한 것이 아님.

**4. 스킬 없는 baseline이 강하다.** 어떤 스킬도 없는 Claude Opus 4.7은 *매우* 능숙한 음악 조언 baseline — GLM-5.1 baseline보다 훨씬 강함. 특히 P5: 스킬 없는 답변이 아름답게 쓰인 곡 초안. 스킬 로딩으로부터의 향상은 따라서 *이미 강한 baseline 위에 더해지는 것* — 양쪽이 모두 컸던 GLM 런보다 더 보수적인 테스트.

## Cross-benchmark 종합

이 파일과 `v1.0-eval.md`는 함께 스킬 효과의 layered 그림을 확립:

| 메커니즘 | 테스트 | 효과 |
|---------|--------|------|
| Approach 1 (시스템 프롬프트 주입, GLM-5.1) | `v1.0-eval.md` | 행동 형성: SKILL.md vs baseline 6/8 블라인드 승. 가장 강한 신호: P8 boundary 거절. |
| Approach 2 (네이티브 lazy loading, Claude Opus 4.7) | 이 파일 | 행동 형성 + 콘텐츠 깊이 잠금 해제: with-skill이 한 번도 outright 안 짐; 명시적 reference-content 인용 (P5 sigimsae 값); 정교한 boundary 처리 (P8); file-structure 인용. |

![Cross-benchmark 승 패턴](./charts/chart_cross_benchmark.png)

이 두 발견을 함께 보면 v1.0 스킬의 가치를 더 깨끗하게 진술할 수 있음:

> **`SKILL.md` 자체만으로도 모델 행동을 스킬의 철학(decision-frame, 장르 인식, boundary 준수)으로 형성하기에 충분하다.** GLM 벤치마크가 확립한 것 — 컨텍스트에 콘텐츠가 존재하면, SKILL.md 포맷에 특별히 튜닝되지 않은 모델에서도 관찰 가능한 행동 변화가 일어남.
>
> **전체 스킬(references와 lazy-loading 메커니즘 포함)은 `SKILL.md` 자체로는 줄 수 없는 콘텐츠 깊이를 잠금 해제한다.** 이 Claude 런이 확립한 것 — 모델이 `genres/korean-traditional.md`를 실제로 읽을 수 있을 때, *yoseong*은 cents 값을 갖는다, generic "ornament" 언급이 아니라.

이건 `v1.0-eval.md` 토론 섹션의 미해결 가설("routing-without-references 문제")을 다룸. 가설은: navigation map만 제공하고 실제 references는 컨텍스트에 없으면 mild dissonance 발생 (C의 leakier P8 boundary에서 가장 명확). 여기서는 references가 tool call로 *접근 가능* — dissonance가 사라지고, 모델의 boundary 처리가 테스트된 모든 조건 중 가장 정교해짐.

## 향후 작업 (이 런 후 개정)

`v1.0-eval` 향후 작업 리스트는 여전히 적용되지만 우선순위가 바뀜:

1. **계측된 Approach 2.** 스킬 로딩 메커니즘에 후킹해 어떤 참조 파일이 어떤 프롬프트에서 접근되는지 로그. Lazy-loading 추론을 확정(또는 반박). Tool calls 모니터링 wrapper 스크립트로 가능.

2. **Claude cross-tier.** 최신 Sonnet 4.6과 Haiku 4.5에 동일한 8 프롬프트를 스킬 로드 상태로 실행. 모델 능력 티어별 효과 매핑 — 스킬이 작은 모델을 *더* 도울까(scaffolding 필요) *덜* 도울까(references navigate 능력 부족)?

3. **분산 체크.** 각 (프롬프트, 조건) 셀을 n=3으로 실행, 셀당 다수결. 생성 노이즈 매끄럽게 함.

4. **Adversarial 프롬프트.** Boundary 메커니즘이 이제 경험적으로 검증됐으니, boundary를 *깨려는* 프롬프트 설계 (예: "믹싱 관련인 거 알지만 어쨌든 도와줘", 또는 작곡으로 위장한 믹싱 질문). Boundary 존재만이 아닌 robustness 테스트.

5. **Cross-language.** 동일 8 프롬프트의 영어 대응판. 문화 특수성 발견(P5)이 언어 간 재현 가능한지 확인(또는 부정).

6. **외부 프롬프트.** 음악 커뮤니티에서 ~20개 실제 사용자 질문 수집. 프롬프트 작가 편향 제거.

## 이 런으로 주장 가능한 것

다가올 커뮤니티 글쓰기를 위해, `v1.0-eval.md`와 이 파일의 조합은 다음 정직한 주장을 뒷받침:

1. **스킬 콘텐츠는 비-Claude 모델에서도 모델 행동을 형성한다** (GLM-5.1, 블라인드, 6/8 승).
2. **Claude에서 네이티브 스킬 로딩으로 조언의 깊이와 정밀도가 의미 있게 증가한다** — 가장 명확한 곳은 문화 특수성 프롬프트, 거기서 스킬은 baseline이 공급할 수 없는 용어와 구현 값을 제공.
3. **스킬의 boundary 메커니즘은 작동한다**, 그리고 lazy 참조 로딩을 갖춘 능숙한 모델에서는 *우아하게* 작동한다 — 비도움이 되지 않으면서 스코프 보존, 사용자를 스킬 안의 구체적 in-scope 파일로 redirect.
4. **스킬의 첫 버전(`v1.0`)은 작동하는 작곡 어드바이저**다 — 아직 포괄적이지는 않지만, `v1.1` 개선(계측된 벤치마크, 더 많은 장르, 더 큰 참조 셋, 평가 주도 반복)으로의 명확한 경로 있음.
