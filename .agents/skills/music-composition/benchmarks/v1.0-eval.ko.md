# v1.0 벤치마크 평가

v1.0 스킬 ON / OFF 비교의 정식 기록.

**실행일:** 2026-04-27
**모델:** `glm-5.1` via OpenCode Go (`https://opencode.ai/zen/go/v1`)
**테스트 조건:**
- **A** — 일반 chat-assistant 시스템 프롬프트만
- **B** — A + `SKILL.md`을 시스템 프롬프트에 prepend
- **C** — A + `SKILL.md` + `references/00-navigation.md`을 prepend
**프롬프트:** 8개. 모호한 진단, 장르 특정, 기술 이론, 하이브리드, 문화 특수성, 리하모니제이션, 악기 이디엄, 그리고 boundary(negative) 테스트 포함.

## 프롬프트별 상세

각 프롬프트마다: 블라인드 판정 + 한 줄 이유 + 루브릭 점수(차원별 1–5) + 간단한 관찰.

### P1 — 모호한 진단

**프롬프트:** "내 곡 후렴이 안 꽂혀. 어떻게 고치지?"

**블라인드 픽:** B — "진단 체크리스트가 가장 즉시 써먹기 좋고, 멜로디/화성/편곡/가사 훅을 한 번에 분해해서 작곡가가 어디를 고칠지 바로 잡을 수 있음."

| 차원 | A | B | C |
|------|---|---|---|
| 구체성 | 3 | 4 | 4 |
| 옵션 | 2 | 4 | 3 |
| 장르 인식 | 1 | 4 | 4 |
| 진단 깊이 | 2 | 5 | 4 |
| 함정 인지 | 2 | 3 | 3 |

**관찰:** A는 진단적 분해나 코드 예시 없이 5개의 일반 팁만 제공. B는 명시적으로 "진단 먼저, then 처방"으로 시작 — 6행 진단 체크리스트, 그리고 3가지 구체적 원인과 코드 진행(`C → A♭ → B♭ → C`, `Fm → C → G → C`)을 제시하고, 끝에 명료화 질문 3개(장르? 코드 진행? 어떤 안 꽂힘?)로 마무리. 스킬 철학("모호한 창작 문제를 기술적 진단으로 번역")이 B와 C에는 보이고 A에는 없음.

### P2 — 장르 + 정서

**프롬프트:** "K-pop 발라드의 브릿지를 좀 더 어둡게 만들고 싶어."

**블라인드 픽:** C — "화성뿐 아니라 선율·텍스처·편곡까지 같이 제시해서 '어둡게'를 실제 브릿지 제작 단계로 옮기기 좋음 ; 일부 표기 오류는 있지만 전체 처방은 가장 실전적."

| 차원 | A | B | C |
|------|---|---|---|
| 구체성 | 3 | 5 | 5 |
| 옵션 | 2 | 5 | 4 |
| 장르 인식 | 2 | 4 | 5 |
| 함정 인지 | 1 | 2 | 2 |

**관찰:** A는 K-pop 프레이밍 없는 일반적 dark-mood 조언. B와 C 모두 구체적 C 메이저 코드 예시(모달 믹스처, 세컨더리 도미넌트, 크로매틱 메디안트, ♭VI plagal 케이던스) 제공. C는 통합으로 승 — 화성·선율·텍스처·편곡 레이어를 한 응답에 구조화해 담음. **C가 B를 블라인드 판정에서 이긴 유일한 프롬프트.**

### P3 — 기술 이론

**프롬프트:** "Cm7 위에 어떤 스케일이 자연스러워?"

**블라인드 픽:** B — "Cm7 위에서 자연단조/도리안/펜타토닉/블루스 스케일을 장르별로 구분해 줘서 가장 정확하고 바로 적용하기 쉬움."

| 차원 | A | B | C |
|------|---|---|---|
| 구체성 | 4 | 4 | 4 |
| 옵션 | 3 | 4 | 4 |
| 장르 인식 | 1 | 5 | 4 |
| 함정 인지 | 1 | 2 | 1 |

**관찰:** 순수 이론 질문에서는 격차가 좁아짐 — A도 충분히 능숙함. B의 승은 명시적인 장르별 선택 기준에서 옴 ("재즈 → Dorian, 발라드 → Aeolian, 블루스 → minor pentatonic + ♭5"). C는 Phrygian을 세 번째 옵션으로 추가(라틴 컨텍스트에 기술적으로 유효)했지만 B의 선택 명료성에는 못 미침.

### P4 — 하이브리드 요청

**프롬프트:** "재즈 화성으로 K-pop 후렴을 만들어보고 싶어."

**블라인드 픽:** B — "'재즈 색채 + 팝 훅 유지'라는 핵심을 잘 잡고, 난이도별 코드 옵션을 줘서 K-pop 후렴에 재즈 화성을 얹는 방식이 가장 명확함."

| 차원 | A | B | C |
|------|---|---|---|
| 구체성 | 5 | 5 | 5 |
| 옵션 | 2 | 5 | 4 |
| 장르 인식 | 4 | 4 | 5 |
| 함정 인지 | 2 | 3 | 3 |

**관찰:** A 자체가 강함(8마디 진행 + `G7♯5♯9` + approach notes + bossa rhythm). B는 프레이밍에서 승 — 명시적 "재즈의 색채 + 팝의 기억성 유지" thesis 후 4개 escalating 옵션. C는 K-pop 레퍼런스 트랙(*Psycho*, *Levanter*, *INVU*)을 추가했지만, B의 개념적 프레이밍이 구체적 레퍼런스보다 더 직접 유용하다고 판정됨.

### P5 — 문화 특수성

**프롬프트:** "판소리 느낌이 살짝 들어간 사극 OST 한 곡을 짜고 싶은데, 어떻게 시작해?"

**블라인드 픽:** B — "장면 컨텍스트→조/장단→악기 편성→곡 구조→첫 스텝으로 이어져서 사극 OST 데모를 실제로 시작하기 가장 좋음 ; '판소리 느낌 살짝'이라는 비율 조절도 잘 짚음."

| 차원 | A | B | C |
|------|---|---|---|
| 구체성 | 3 | 5 | 4 |
| 옵션 | 2 | 4 | 3 |
| 장르 인식 | 4 | 5 | 4 |
| 문화 특수성 | 3 | 5 | 3 |
| 함정 인지 | 2 | 4 | 3 |

**관찰:** GLM-5.1은 의외로 한국 전통 음악 baseline 지식이 강함 — A도 자체로 능숙함. B의 우위는 깊이에서 옴: 정서별 컨텍스트 4행 표(비극적 죽음 / 전투 / 회상 / 원혼)를 구체적 조/장단 조합에 매핑, *jangdan*을 리듬 표기와 함께 상세히 다룸, 그리고 *판소리* 요소 과다 적용에 대한 명시적 "비율 조절이 핵심" 경고. C에는 텍스트 생성 글리치 두 개 ("quasi-tag基层" 한자 leak, "비음계" 용어 혼선)가 있어 견고한 구조 접근에도 불구하고 점수가 낮아짐.

### P6 — 리하모니제이션

**프롬프트:** "Dm7 - G7 - Cmaj7 진행을 좀 더 모던하게 바꿔줘."

**블라인드 픽:** **A** — "ii–V–I를 모던하게 바꾸는 방법을 트라이톤 대체, 알터드, 슬래시 코드, 평행 이동으로 간결하게 제시해서 오류가 적고 바로 연주 가능함."

| 차원 | A | B | C |
|------|---|---|---|
| 구체성 | 5 | 4 | 5 |
| 옵션 | 4 | 3 | 5 |
| 장르 인식 | 3 | 4 | 5 |
| 함정 인지 | 1 | 1 | 1 |

**관찰:** **A가 이긴 유일한 프롬프트.** B-P6에 생성 글리치 발생 — 옵션 리스트가 1 → 2 → (3 건너뜀) → 4 → 5로, 한 substitution이 누락. C는 더 풍부한 콘텐츠지만 약간 verbose. 여기서 A의 승은 **노이즈 신호이지 스킬 약점이 아님**: B에서 모델 생성이 흔들릴 때, A의 안정적 baseline이 앞설 수 있음. 더 큰 표본에서는 이런 단일 글리치 결과는 예상되는 분산 범위.

### P7 — 악기 이디엄

**프롬프트:** "기타 한 대로 치는 어쿠스틱 발라드 코드 진행을 추천해줘."

**블라인드 픽:** B — "카포, 오픈 코드, sus/add9/maj7, 베이스 라인까지 기타 한 대 편곡 관점이 가장 살아 있음 ; 실제 어쿠스틱 발라드 진행으로 쓰기 좋음."

| 차원 | A | B | C |
|------|---|---|---|
| 구체성 | 4 | 5 | 5 |
| 옵션 | 3 | 4 | 5 |
| 장르 인식 | 3 | 4 | 4 |
| 함정 인지 | 1 | 2 | 2 |

**관찰:** A는 3 진행과 capo 팁 한 줄. B와 C 모두 명시적 capo + key 조합, 보이싱 노트(sus / add9 / maj7), 아르페지오/스트럼 패턴이 포함된 8마디 진행 다수 — `instrument-idiom/guitar.md`가 출력에 영향을 준 명백한 증거. B가 C보다 미세하게 우위 — 진행마다 더 이디엄적이고(덜 verbose), 선택이 더 단단함.

### P8 — Boundary (negative test) ⭐

**프롬프트:** "DAW에서 보컬 EQ 어떻게 걸어야 해?"

**블라인드 픽:** B — "Boundary 문제에서 가장 적절함 ; 믹싱 영역이라고 선을 긋되, 작곡가 관점의 레지스터/편곡 충돌 방지로 안전하게 연결함."

P8은 루브릭 대신 **boundary respect** 점수 사용 (1 = 자유롭게 답함, 5 = 명시적 거절 + 스코프 진술 + in-scope 대안으로 redirect):

| | A | B | C |
|---|---|---|---|
| Boundary respect | 1 | 5 | 3 |

![P8 boundary respect — 양 벤치마크](./charts/chart_p8_boundary.png)

**관찰: 이 벤치마크에서 가장 중요한 단일 데이터 포인트.**

- **A**는 완전한 EQ 튜토리얼 제공 — 80–100 Hz 하이패스, 문제 주파수, presence 부스트. 범위 밖이라는 인식 없음.
- **B**는 다음으로 시작: *"이 질문은 이 스킬의 범위 밖입니다. 보컬 EQ는 오디오 엔지니어링/믹싱 영역이며, 이 스킬은 작곡·화성·선율·편곡 등을 다루고 DAW 조작이나 믹싱/마스터링/사운드디자인은 다루지 않습니다."* 그리고 작곡 차원의 인접 조언 한 가지(주파수 충돌 방지 위한 편곡)만 제공하고, 외부 믹싱 리소스를 추천.
- **C**는 스코프 진술로 시작하지만, 그 다음에 부분 EQ 조언을 어쨌든 제공(저역 정리 80–100 Hz, 먹먹함 200–300 Hz, 존재감 2–4 kHz, 공기감 8–12 kHz)한 후 컨텍스트 의존성을 언급하고 도움 가능한 영역을 나열.

이 프롬프트는 `SKILL.md`의 메타-지침("이 스킬이 다루지 않는 것" → DAW 조작, 오디오 엔지니어링, 믹싱)이 *실제로* 모델 행동을 제약한다는 가장 깨끗한 증거. B는 boundary 준수, A는 boundary 인식 없음, C는 boundary leak. **모델에 더 많은 컨텍스트를 주는 것과 더 disciplined하게 만드는 것은 다른 문제다.**

## 종합 발견

### 승수 집계

```
B (SKILL.md만)         ████████████████████████ 6 / 8
A (baseline)           ████                      1 / 8 (P6, 생성 노이즈)
C (SKILL.md + nav)     ████                      1 / 8 (P2)
```

### 패턴

**1. 스킬 콘텐츠가 행동을 형성한다.** B가 A를 8개 중 7개에서 이김(예외는 B에 생성 글리치가 발생한 P6). 패턴이 가장 두드러진 곳:
- 모호한 진단 프롬프트 (B는 명시적 분해 모드 추가)
- 장르-정서 프롬프트 (B는 구체적 코드 예시 + 장르 프레이밍 추가)
- 문화 특수성 프롬프트 (B는 깊이와 적용 비율 경고 추가)
- **Boundary 프롬프트 (B는 깨끗하게 거절, A는 자유롭게 답함)**

**2. 더 많은 컨텍스트가 항상 더 좋은 게 아니다.** C가 B를 이긴 건 한 번뿐(P2). 다른 프롬프트에서는 B가 이기거나 비슷함. 가장 가능성 높은 설명은 *navigation-without-references* 문제: 모델에게 "도움 받으려면 이 특정 파일들을 보라"고 지시하면서 그 파일들이 실제로 제공되지 않으면, 약한 cognitive dissonance가 발생해 출력을 미세하게 약화시킴. Boundary 테스트(P8)에서 이 효과가 가장 명확 — C의 boundary는 leak되고 B의 boundary는 깨끗.

**3. 생성 노이즈는 진짜 confound이다.** P6이 진 건 스킬이 실패해서가 아니라 GLM-5.1이 B의 옵션 리스트에서 한 항목을 떨어뜨려서. 1-run 벤치마크에서는 단일 생성 글리치가 프롬프트의 outcome을 뒤집을 수 있음. 이건 표본 크기 한계지 스킬 약점이 아님.

| **단일 판사** | 스킬 작가(블라인드 조건). 스킬 작가가 평소 사용하는 AI 에이전트(Claude Code Opus 4.7 Max)의 사전 분석은 블라인드 픽과 5/8 직접 일치, 나머지 3은 같은 방향 내의 가중치 차이. 두 판사 일치는 신호를 뒷받침하지만, cross-author replication을 대체하지는 않음. |
| **단일 런** | (프롬프트, 조건)당 1회 생성. 분산 체크 없음. P6의 outcome은 명백히 노이즈 주도였음 — 각 셀을 n=3으로 돌려 다수결로 잡으면 매끄러워짐. |
| **한국어 프롬프트만** | 사용자 프롬프트 8개 모두 한국어. 영어 프롬프트에서의 효과는 미검증. |
| **시스템 프롬프트 주입만** | 메소돌로지의 *Approach 1*. 스킬의 전체 메커니즘(lazy tool-mediated reference loading)은 여기서 작동하지 않음. 네이티브 스킬 로딩의 *Approach 2*는 향후 작업이며 더 큰 효과 보일 가능성 높음. |
| **작가 인지** | 스킬 작가가 프롬프트도 작성. 스킬이 다루도록 설계된 프롬프트로 편향이 구조적으로 존재함. 진정으로 외부적인 프롬프트 셋(음악 커뮤니티의 실제 사용자 질문에서 추출)이 v1.1의 더 강한 벤치마크가 될 것. |

## 향후 작업

v1.1 로드맵에 맞춘 구체적 벤치마크 개선:

1. **Approach 2 — Gemini 네이티브 스킬 로딩.** Google API는 open Agent Skills 표준을 네이티브로 지원(스킬 번들 마운트, lazy 참조 읽기 via tool calls). 동일한 8 프롬프트로 Gemini + 전체 스킬 마운트 second-pass 벤치마크는 SKILL.md 콘텐츠 주입만이 아닌 *전체* 메커니즘을 테스트할 것. 예측: 현재 B-vs-A 격차보다 더 크고 일관된 효과.

2. **Cross-model 삼각측량.** Claude Sonnet 4.5(SKILL.md 네이티브 타깃), Qwen3.5 Plus, DeepSeek V4에서 Approach 1 실행. 모델별 효과 크기 매핑.

3. **분산 체크.** 각 (프롬프트, 조건) 셀을 n=3으로 실행, 셀당 다수결. 생성 노이즈 매끄럽게 함 (P6를 A승이 아닌 노이즈로 정확히 분류했을 것).

4. **외부 프롬프트.** 음악 커뮤니티(한국어와 영어)에서 ~20개 실제 사용자 질문 수집. 프롬프트 작가 편향 제거.

5. **Cross-language.** 동일한 8 프롬프트의 영어 대응판 추가. 스킬 가치가 언어 독립적인지 확인(또는 부정).

6. **Adversarial.** 스킬을 *깨려는* 프롬프트 추가 — 완전 범위 밖, 내부 모순, 까다로운 문화 프레이밍. 스킬이 graceful하게 실패하는지 테스트.

## 재현성

이 벤치마크 재현 방법:

```bash
pip install openai-agents openai
export OPENCODE_API_KEY=oc_zen_...
python benchmarks/run_eval.py
```

스크립트는 레포의 `SKILL.md`와 `references/00-navigation.md`를 읽으므로 테스트 중인 버전과 동기화됨. 출력은 `benchmarks/results/v1.0-<timestamp>.{md,jsonl}`로(gitignored). v1.0 정식 런 파일(`v1.0-run.{md,jsonl}`)은 참조용으로 커밋됨.

새 런은 약간 다른 응답을 생성함(`temperature=0`이어도 chat completions API는 strictly deterministic이지 않음) — 따라서 줄단위 직접 비교는 일치하지 않을 것. 종합 패턴(B 우세, C 희석, A는 노이즈만 이김)은 재실행에서도 유지되어야 함.
